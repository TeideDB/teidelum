THE PROBLEM

AI agents deserve better than file scanning

Most "knowledge base" setups point an LLM at a folder of files. That works for small personal vaults—but it breaks down fast. No ranking. No structure. No relationships. Every query re-reads everything.

Without infrastructure
  • LLM scans raw files on every query
  • No relevance ranking—all content is equal
  • Can't aggregate, filter, or join data
  • No entity relationships—flat documents only
  • Locked to one tool, one LLM, one format
  • Slows down as knowledge grows
With Teidelum
  • Indexed BM25 search with fuzzy matching
  • Ranked results with source attribution
  • SQL analytics across all structured data
  • Graph traversal over entity relationships
  • MCP for any AI agent, REST for any app
  • Stays fast regardless of data volume
WHAT YOU GET
01

Indexed Search

BM25 ranking with fuzzy matching and highlighted snippets. Your AI agent asks a question and gets the 5 most relevant results—not a token-heavy dump of every file that might be related.

02

SQL Analytics

Run columnar queries over structured data from any synced source. Filter, group, sort, and join—things that are impossible when your knowledge base is just a pile of markdown files.

03

Graph Traversal

Foreign-key relationships between entities create a navigable graph. Ask "what's connected to this customer?" and get answers—no need to teach your LLM a custom schema on every conversation.

04

Universal API

MCP over stdio for AI agents. REST over HTTP for applications and scripts. Same data, same capabilities, two transports. Connect Claude, GPT, your internal dashboard, or a cron job—all at once.

05

Work Tool Sync

Pull from Notion pages, Zulip messages, and kdb+ databases. Incremental sync means only changed records on each run. Your knowledge base stays current without rebuilding from scratch.

06

Single Binary, Zero Ops

One Rust binary. No Docker. No database server. No config files. Set a data directory and start it. Everything runs locally—your organizational knowledge never touches a third-party cloud.

SEE IT IN ACTION

Two lines to start, one API to use

Run the binary, point your MCP client at it, or call the REST API from any language. Search, query, and traverse in seconds.

terminal 
# Start with HTTP server on port 8080
teidelum --port 8080

# Search across all indexed content
curl -X POST http://localhost:8080/api/v1/search \
  -H "Content-Type: application/json" \
  -d '{"query": "JWT token rotation", "limit": 5}'

# Run SQL over structured data
curl -X POST http://localhost:8080/api/v1/sql \
  -H "Content-Type: application/json" \
  -d '{"query": "SELECT assignee, count(*) FROM tasks WHERE status = '\''open'\'' GROUP BY assignee"}'

# Traverse entity relationships
curl -X POST http://localhost:8080/api/v1/graph/neighbors \
  -H "Content-Type: application/json" \
  -d '{"start_table": "customers", "start_id": "42", "max_depth": 2}'
mcp-agent.jsonc 
// AI agents call the same operations via MCP tools

// Search
{ "tool": "search", "params": { "query": "auth migration plan", "limit": 5 } }

// SQL
{ "tool": "sql", "params": { "query": "SELECT * FROM projects WHERE deadline < '2026-04-01'" } }

// Create a table (write tools work on both MCP and REST)
{
  "tool": "create_table",
  "params": {
    "name": "meeting_notes",
    "source": "agent",
    "columns": [
      { "name": "id", "type": "int" },
      { "name": "title", "type": "varchar" },
      { "name": "date", "type": "varchar" }
    ],
    "rows": [[1, "Sprint Planning", "2026-03-01"]]
  }
}
